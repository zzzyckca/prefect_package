The Business Need

  • × Legacy Windows Task Scheduler is brittle, hard to monitor, and prone to "silent failures".
  • ✓ Data Pipelines require robust error handling, immediate email alerts, and clear, accessible logs.
  • ✓ We need 100% data privacy: A completely local, open-source orchestration system that doesn't rely on the cloud.

The Architectural Solution

Isolated Environment

The entire system lives securely in C:\prefect. The `install.bat` wrapper mathematically builds the environment from scratch, ensuring perfect reproducibility without affecting the host machine.

Invisible Operation

start.bat leverages native Windows VBScript to boot the Prefect Server and Worker instances into "Session 0", meaning absolutely no intrusive console windows on your taskbar.

Invisible Background Operations

The system is designed to run silently without any popup Command Prompt windows interrupting users.

# Inside start.bat
echo Set WshShell = CreateObject("WScript.Shell") > run_hidden.vbs
echo WshShell.Run chr(34) ^& "prefect server start" ^& chr(34), 0 >> run_hidden.vbs
cscript //nologo run_hidden.vbs

Window Style: 0

The 0 parameter triggers "wsHide" in the Windows API, instantly sending the process to the background.

Ghost Processes

If you need to restart the machine or stop Prefect, you run stop.bat which cleanly kills these hidden processes.

Step 1: Declare It

Step 2: Install It

User Guide: Replacing a Script

To schedule a brand new Python script to run, you don't need to write any Prefect code. You just modify the prefect.yaml list to point to your new file.

parameters:
  scripts_to_run:
    - 'c:\temp\old_job.py'
    - 'c:\temp\my_new_job.py' <-- Swap the filename

Once you change the target script, double-click deployment.bat and Prefect will automatically run the new file instead!

We control pipelines entirely through a simple `prefect.yaml` file, unlocking enterprise-grade precision:

Standard daily or weekly execution patterns.

Intervals

RRule

User Guide: Custom Triggers

Instead of Cron, you can use advanced Prefect Triggers in the YAML:

1. The Interval Trigger

interval: 1209600.0 anchor_date: "2026-03-02T09:00:00Z"

Runs exactly every 14 days down to the second, using the anchor date to lock the loop to a specific starting point.

2. The Business Day Hook

target_business_day: 5

Pass this as a parameter, and the Master Pipeline blocks execution until the actual 5th Toronto work day.

Python Basics vs. Prefect Engine

def function()

Inside Prefect, standard Python perfectly executes! But it acts as a "black box". The UI won't track its logs, duration, or retries.

def do_work():
    print("Invisible to UI Dashboard")
    1 / 0 # Crashes permanently

@task

A monitored step. Prefect records how long it took and captures all print logs/errors into the UI database.

@task(retries=2)
def do_work():
    print("Saved to UI log!")
    1 / 0 # Safely pauses, then retries

@flow

The Orchestrator root. Only a @flow can be scheduled in prefect.yaml.

@flow(name="Main")
def pipeline():
    do_work() # Calls the @task
    print("Flow finished!")

The Triggering Mechanism

1. The Server & Worker

start.bat starts two background processes. The Server holds the schedules. The Worker constantly polls the Server asking, "Is it time yet?"

2. The YAML Schedule

When you edit prefect.yaml, you are simply declaring a schedule. But Prefect doesn't know about it yet.

3. The Deployment Trigger

When you double click deployment.bat, it runs:
prefect deploy --all

This securely transmits the YAML schedule to the live Server. The Server then tells the Worker exactly when to trigger the Master Pipeline.

Adding Jobs (Manager's Guide)

Zero Prefect code knowledge is required to add or modify a pipeline.

  1. Write standard Python code (e.g., c:\temp\script.py).
  2. Open c:\prefect\prefect.yaml.
  3. Add the script's path to the scripts_to_run list.
  4. Set the desired cron or target_business_day variable.
  5. Double-click deployment.bat to push live!

User Guide: Sequential Workflows

Do you have a pipeline where Script B absolutely must wait for Script A to finish? Prefect handles this natively.

Simply list them in order. The Master Pipeline will run them sequentially and protect the downstream jobs if an upstream job fails.

parameters:
  scripts_to_run:
    - 'c:\temp\A_Extract.py' # Runs First
    - 'c:\temp\B_Transform.py' # Runs Second
    - 'c:\temp\C_Load.py' # Runs Third

Security & Data Privacy

This entire architecture is built as an Air-Gapped Local Implementation.

  • Zero Cloud Dependency: We do not use Prefect Cloud. We run the open-source Prefect Server natively right on our own Windows machine.
  • Total Data Sovereignty: Your company's sensitive data, API keys, and pipeline logic never leave the internal network.

Under the Hood: @task & @flow

The core engine is master_pipeline.py.

In Prefect, a @flow is the main orchestrator (the container). A @task is an individual, trackable step within that container.

  • @task: run_task() encapsulates each of your raw .py scripts so Prefect can monitor them.
  • @task: check_toronto_business_day() runs the mathematical holiday checks.
  • @flow: run_script_list() strings them all together sequentially.
master_pipeline.py
@task
def run_task(script_path):
    result = subprocess.run(["python", script_path])
    # Inherits pure python exit statuses

@flow(name="Master-Pipeline-Runner")
def run_script_list(scripts_to_run: list[str]):
    for script in scripts_to_run:
        run_task(script)

User Guide: New Workflows

Need a weekend script that runs on a totally different schedule? Add a completely new deployment block to the bottom of prefect.yaml.

# Just copy and paste the block below the others!
- name: "weekend-report-deployment"
  entrypoint: "master_pipeline.py:run_script_list"
  parameters:
    scripts_to_run:
      - 'c:\temp\weekend_script.py'
  schedules:
    - cron: "0 12 * * 6,0" # Sat/Sun at Noon

Double-click deployment.bat and your new Weekend Flow will appear in the UI.
